## ACP Phase Transitions

- `READY -> IN_PROGRESS`: dependencies and source-merge gate verified; instruction commit recorded before edits.
- `IN_PROGRESS -> EVIDENCE_READY`: scoped implementation and all required local tests complete.
- `EVIDENCE_READY -> WAITING_FOR_PR_REVIEW`: PR exists; transport validation confirms URL, head SHA, changed-file list, CI runs and evidence are accessible; one final Gitea evidence comment is posted.
- `WAITING_FOR_PR_REVIEW -> IN_REVIEW -> COMPLETED`: independent reviewer evaluates the exact head SHA; completion requires a SHA-bound accepting verdict, green required CI, merge by Eddie/operator, and ticket closure.
- Any pre-completion state may become `BLOCKED`; record the category and stop rather than inventing a workaround. `CHANGES_REQUESTED` returns to `IN_PROGRESS` with prior findings retained.

ACP `v0.1.0-draft` has no normative state machine in `SPEC/`. These are the repository control states required by `Homelab/Architecture/AGENTS.md` and accepted Pilot 57 findings; they MUST NOT be represented as a new ACP specification.

## Required ACP Evidence

The final Gitea evidence comment MUST contain:

- `instruction_file`, `instruction_commit`, `instruction_reloaded`, `pr_head_commit`, `changed_files`, `ci_run_ids`, and `review_target: pr` exactly as required by `Homelab/Architecture/AGENTS.md`.
- Base commit, branch, Gitea PR URL, exact test commands and outcomes, required CI job names/run URLs, and an acceptance-criterion-to-evidence matrix.
- Declaration of whether runtime, deployment, DNS, secrets, accounts, containers or external systems were mutated.
- Rollback statement and the complete findings record.
- Transport validation: PR head equals the evidenced SHA; changed files equal the permitted scope; CI and evidence links resolve for the reviewer.

Approval is valid only for the evidenced head SHA. A later push invalidates it and requires re-review.
